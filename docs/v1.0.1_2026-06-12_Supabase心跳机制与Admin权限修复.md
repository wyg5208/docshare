## 任务目标

本次迭代主要解决三个生产环境问题：
1. Admin 用户登录后无法访问 `/admin` 后台，被错误重定向到首页
2. Dashboard 导航图标消失（与问题1同源）
3. Supabase 免费套餐项目因无活动自动休眠，导致服务不可用

同时顺带完善了邮箱验证流程和中间件安全性。

## 问题诊断与分析

### 问题 1: Admin 登录后无法访问 /admin，被重定向到首页

**现象**: 使用 admin 账号成功登录后，访问 `/admin` 路由时被中间件重定向到首页 `/`，同时 Header 组件中的 Dashboard 按钮也消失了。

**原因**: `middleware.ts` 中对 admin 路由有角色校验逻辑——查询 `profiles` 表的 `role` 字段，如果不是 `"admin"` 则重定向到首页。而 admin 用户在 `profiles` 表中的 `role` 字段值为默认的 `"viewer"`，并非 `"admin"`。Header 组件同样基于 `profile?.role === "admin"` 判断是否显示 Dashboard 按钮，因此按钮也一并消失。

**根因**: Seed 脚本 (`scripts/seed-users.ts`) 未被正确执行或执行后数据被覆盖，导致 admin 用户的角色数据不正确。

### 问题 2: Dashboard 图标不见了

**现象**: Header 导航栏中管理员专属的 Dashboard 按钮不再显示。

**原因**: 与问题1同源。Header 组件通过 `profile?.role === "admin"` 控制 Dashboard 按钮的可见性，由于 role 字段值不正确，按钮自然不渲染。

**解决**: 修复问题1后，Dashboard 图标自动恢复。

### 问题 3: Supabase 免费套餐项目自动休眠

**现象**: 部署在 Vercel 上的应用突然无法正常工作，所有 Supabase 请求超时或返回错误。

**原因**: Supabase 免费套餐（Free Plan）有自动暂停机制——如果项目在 7 天内没有任何活动（数据库查询、API 请求等），项目会自动进入暂停状态，需要手动到 Supabase Dashboard 恢复。

**影响**: 生产环境不可用，用户无法正常访问任何功能。

## 解决方案与实施

### 1. Admin 角色修复

**方案**: 重新运行 seed 脚本修复 admin 用户的角色数据。

```bash
npx tsx scripts/seed-users.ts
```

此脚本会将 admin 用户在 `profiles` 表中的 `role` 字段正确设置为 `"admin"`，从而同时解决后台访问和 Dashboard 图标两个问题。

### 2. Supabase 心跳机制（核心新增功能）

**方案**: 实现自动化心跳机制，通过定时任务定期向 Supabase 发送轻量查询，防止项目因无活动而被暂停。

#### 2.1 心跳 API 端点

新建 `src/app/api/health/route.ts`，实现 `GET /api/health` 接口：

- 对 Supabase 执行轻量级 `HEAD` 查询（`select("id", { count: "exact", head: true })`），仅获取记录数，不传输实际数据
- 支持 Bearer Token 认证（通过 `CRON_SECRET` 环境变量），同时允许手动无认证健康检查
- 返回状态信息：`ok`（正常）、`degraded`（查询出错）、`error`（异常）
- 包含延迟指标 `latency_ms`，便于监控

#### 2.2 Vercel Cron Job 配置

新建 `vercel.json`，配置 Cron Job 每天 UTC 00:00 自动调用心跳接口：

```json
{
  "crons": [
    {
      "path": "/api/health",
      "schedule": "0 0 * * *"
    }
  ]
}
```

#### 2.3 环境变量

在 `.env.local` 和 `.env.example` 中添加 `CRON_SECRET` 变量：

```env
# Cron Job secret (Vercel sets this as Authorization header for cron invocations)
# Generate with: openssl rand -hex 32
CRON_SECRET=your-random-secret-here
```

### 3. 中间件优化

**方案**: 在 `src/lib/supabase/middleware.ts` 中添加 API 路由快速通道，跳过认证检查。

**原因**: 心跳接口 `/api/health` 和其他 API 路由不应经过中间件的认证流程（它们各自处理认证逻辑），跳过中间件认证可以让 API 路由更快、更稳定。

```typescript
// Skip auth checks for API routes (they handle their own auth)
if (request.nextUrl.pathname.startsWith("/api")) {
  return supabaseResponse;
}
```

同时新增邮箱验证门控逻辑：未验证邮箱的用户只能访问 `/verify-email`、`/login`、`/register`、`/forgot-password` 等特定路由。

### 4. 登录/注册流程邮箱验证增强

#### 4.1 登录页 (`login/page.tsx`)
- 登录成功后检查用户邮箱验证状态
- 未验证邮箱的用户被引导到 `/verify-email` 页面，并提示 "Please verify your email to continue."

#### 4.2 注册页 (`register/page.tsx`)
- 注册时添加 `emailRedirectTo` 参数，指向 `/api/auth/callback`
- 注册成功后引导用户到 `/verify-email` 页面（而非之前的 `/login`）
- 更新提示信息为 "We've sent a verification link to your email."

## 技术实现细节

### 心跳 API 核心代码

```typescript
// src/app/api/health/route.ts
export async function GET(request: NextRequest) {
  const authHeader = request.headers.get("authorization");
  const cronSecret = process.env.CRON_SECRET;

  if (cronSecret && authHeader !== `Bearer ${cronSecret}`) {
    // Allow unauthenticated access for manual health checks
  }

  const start = Date.now();
  const supabase = await createClient();

  // Lightweight query to keep Supabase active
  const { error } = await supabase
    .from("categories")
    .select("id", { count: "exact", head: true });

  if (error) {
    return NextResponse.json(
      { status: "degraded", error: error.message, latency_ms: Date.now() - start },
      { status: 503 }
    );
  }

  return NextResponse.json({
    status: "ok",
    latency_ms: Date.now() - start,
    timestamp: new Date().toISOString(),
  });
}
```

### Vercel Cron Job 配置

```json
{
  "crons": [
    {

    }
  ]
```

> **说明**: `schedule` 使用标准 Cron 表达式，`0 0 * * *` 表示每天 UTC 00:00 执行一次。Vercel Hobby 计划支持每天最多执行一次 Cron Job。

## 经验教训

### 1. Supabase 免费套餐休眠机制

Supabase Free Plan 项目会在 **7 天无活动后自动暂停**。这是一个容易被忽视但影响严重的问题。对于生产环境，必须：
- 配置心跳机制保持项目活跃
- 考虑升级到 Pro Plan（$25/月）以获得 "never pause" 保障
- 设置外部监控（如 UptimeRobot）及时感知服务中断

### 2. Admin 角色管理

- Seed 脚本是初始化数据的重要工具，但要确保其幂等性（可重复执行）
- 生产环境中 admin 角色的设置应有更安全的机制（如 Supabase Dashboard 手动设置或专用管理接口）
- 角色变更应有审计日志

### 3. 中间件设计原则

- API 路由应该在路由层面自行处理认证，中间件不应重复认证逻辑
- 心跳等运维接口需要快速响应，不能被复杂的认证流程拖慢
- 邮箱验证等门控逻辑应该放在中间件层统一处理，而非分散在各个页面

### 4. 环境变量管理

- 新增环境变量时，务必同步更新 `.env.example` 模板文件
- 生产环境（Vercel Dashboard）也需要同步配置对应的环境变量
- 敏感信息（如 `CRON_SECRET`）应使用安全的随机生成方式

## 总结

本次迭代以 **运维稳定性** 和 **权限修复** 为核心：

1. **权限修复**：通过重新执行 seed 脚本修复了 admin 用户的角色数据，恢复了后台访问和 Dashboard 导航功能。这是一个数据问题而非代码问题，提醒我们在部署流程中需要更可靠的数据初始化机制。

2. **心跳机制**：针对 Supabase 免费套餐的自动休眠问题，实现了基于 Vercel Cron Job 的心跳机制。该方案轻量、自动化、零成本，能有效防止生产环境因数据库休眠而中断服务。

3. **中间件优化**：为 API 路由添加了快速通道，避免不必要的认证检查；同时增强了邮箱验证的门控逻辑，提升了系统安全性。

4. **流程完善**：登录和注册流程增加了邮箱验证状态检查，确保未验证用户被正确引导，提升了用户体验和系统安全性。
