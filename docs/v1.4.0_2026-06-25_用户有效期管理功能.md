## 任务目标

管理员需要能够设置用户的使用有效期，实现两种有效模式：
1. **期间有效** — 指定起止日期（如 6 月 27 日至 6 月 30 日），期间内用户可正常登录访问
2. **永久有效** — 长期有效，不设时间限制

同时需要确保：
- 手动禁用（`is_active=false`）优先级高于时间期限设置
- 激活状态下有效期自动生效，过期自动禁止登录
- 非激活用户登录时显示友好提醒，引导联系管理员

## 问题诊断与分析

### 问题 1: 原有用户状态模型过于简单
**现象**: `profiles` 表仅有 `is_active` 布尔字段，无法表达时间维度的权限控制需求。
**原因**: 初版设计仅考虑"激活/禁用"两种静态状态，缺少 `valid_from` / `valid_until` 时间字段。

### 问题 2: 登录流程缺少用户状态检查
**现象**: 被管理员禁用的用户仍然可以正常登录和访问系统。
**原因**: 登录页面仅检查邮箱验证状态（`email_confirmed_at`），未检查 `is_active` 和有效期字段。

### 问题 3: 中间件未拦截过期用户
**现象**: 已过期或被禁用的用户在已有 session 的情况下仍能访问受保护路由。
**原因**: `middleware.ts` 的 `updateSession()` 函数仅检查认证状态和 admin 角色，未检查用户的 `is_active` 和有效期。

## 解决方案与实施

### 1. 数据库层

#### 新增迁移文件 `006_user_validity_period.sql`
- 在 `profiles` 表新增两个 `TIMESTAMPTZ` 字段：
  - `valid_from` — 有效期起始时间（`NULL` 表示无起始限制）
  - `valid_until` — 有效期截止时间（`NULL` 表示无截止限制）
- 新增辅助函数 `is_user_effectively_active(user_id UUID)` — 综合判断用户是否实际有效
- 字段逻辑说明：
  | `valid_from` | `valid_until` | 含义 |
  |---|---|---|
  | NULL | NULL | 永久有效 |
  | 有值 | 有值 | 指定期间内有效 |
  | 有值 | NULL | 从某日起永久有效 |
  | NULL | 有值 | 直到某日前有效 |

### 2. 后端 API

#### 修改 `/api/users` PATCH 端点
- 新增 `action: "set_validity"` 动作类型
- 支持传入 `validFrom` 和 `validUntil` 日期字段
- 使用 `createAdminClient()` 绕过 RLS 策略更新 `profiles` 表

```typescript
// 请求体结构
{
  userId: string,
  action: "set_validity",
  validFrom: string | null,  // ISO 日期字符串
  validUntil: string | null   // ISO 日期字符串
}
```

### 3. 中间件层

#### 修改 `src/lib/supabase/middleware.ts`
- 在受保护路由检查中新增用户状态与有效期校验
- 检测逻辑：
  1. `is_active = false` → 强制登出并重定向到 `/login?inactive=true`
  2. `valid_from > now()` 或 `now() > valid_until` → 强制登出并重定向到 `/login?expired=true`
- 缓存 `cachedProfile` 避免 admin 路由角色检查的重复数据库查询

### 4. 登录流程

#### 修改 `src/app/(auth)/login/page.tsx`
- 登录成功后新增用户状态检查步骤：
  1. 查询 `profiles.is_active`, `valid_from`, `valid_until`
  2. 若 `is_active = false`：强制登出 + 显示黄色警告提示
  3. 若不在有效期内：强制登出 + 显示黄色警告提示
- 支持从中间件重定向过来时（`?inactive=true` / `?expired=true`）直接显示对应提示
- 使用 `useSearchParams` 读取 URL 参数，替代原有的 `window.location.search`

### 5. Admin 用户管理页面

#### 修改 `src/app/(admin)/admin/users/page.tsx`
- 表格新增 **Time Permission** 列，使用颜色徽章一目了然展示状态：

| 状态 | 徽章颜色 | 说明 |
|------|---------|------|
| Permanent | 🟢 绿色 | 永久有效，无时间限制 |
| Valid | 🔵 蓝色 | 在有效期内，附显示日期范围 |
| Expired | 🟠 橙色 | 已过期，附显示日期范围 |
| Disabled | 🔴 红色 | 被管理员手动禁用 |

- 点击日历图标按钮可打开内联编辑器：
  - 下拉选择 **Permanent**（永久）或 **Time Period**（期间）
  - 期间模式下显示起止日期选择器
  - 保存后通过 `/api/users` PATCH 接口写入数据库

## 技术实现细节

### 有效期优先级逻辑
```
is_active = false  →  用户始终被禁用（最高优先级）
     ↓ (is_active = true)
valid_from / valid_until 均为 NULL  →  永久有效
     ↓
检查 valid_from ≤ now() ≤ valid_until  →  有效/过期
```

### 中间件缓存优化
为避免 admin 路由角色检查的重复 DB 查询，引入 `cachedProfile` 变量复用首次查询结果：

```typescript
let cachedProfile: { is_active: boolean; valid_from: string | null; valid_until: string | null; role: string } | null = null;
if (isProtected && user) {
  const { data: userProfile } = await supabase
    .from("profiles")
    .select("is_active, valid_from, valid_until, role")
    .eq("id", user.id)
    .single();
  cachedProfile = userProfile;
  // ... validity checks
}

// Admin role check reuses cached profile
if (request.nextUrl.pathname.startsWith("/admin") && user) {
  const profile = cachedProfile || (await supabase
    .from("profiles").select("role").eq("id", user.id).single()).data;
  // ...
}
```

### TypeScript 类型扩展
在 `Database` 接口的 `profiles` 表 Row/Insert/Update 类型中新增：
```typescript
valid_from: string | null;
valid_until: string | null;
```

## 经验教训

### 1. 中间件查询优化
在 `middleware.ts` 中对同一用户的 `profiles` 表多次查询可以通过 `cachedProfile` 变量避免重复 DB 调用。对于 admin 路由，首次查询已包含 `role` 字段，无需二次查询。

### 2. 登录流程的状态检查顺序
用户状态检查需要遵循严格的优先级链：邮箱验证 → 激活状态 → 有效期。任一环节失败都需要先 `signOut()` 再显示提示，防止用户利用残留 session 绕过限制。

### 3. 日期存储边界处理
前端使用 `new Date(validUntil + "T23:59:59")` 将截止日期设为当天 23:59:59，确保截止日当天仍可正常使用（而非 00:00:00 导致当天一早即失效）。
