# 迭代记录_2026-06-29_文档预览权限绕过修复

## 基本信息
- **版本**: v1.9.2
- **日期**: 2026-06-29
- **类型**: Bug 修复（安全性）
- **涉及模块**: 文档预览组件（`document-preview.tsx`）
- **状态**: ✅ 已完成

## 任务目标

修复 `document-preview.tsx` 组件中存在的权限绕过漏洞。该组件在处理不可预览的文件类型时，直接渲染了一个未经权限校验的下载链接，导致仅有 "Only View" 权限的用户也能绕过 `canDownload` 权限控制直接下载文件。修复后统一通过受权限控制的 Download 按钮执行下载操作。

### 核心目标
1. 移除 fallback 分支中绕过权限的直接下载链接
2. 替换为友好的"无法预览"提示，引导用户使用受权限控制的主操作路径
3. 清理因移除下载链接而产生的未使用导入

## 问题诊断与分析

### 问题 1: 文档预览 fallback 分支权限绕过

**现象**：
- 当用户访问不支持在线预览的文件类型（如 `.docx`、`.xlsx` 等非标准类型）时，页面会渲染一个"Download to view"按钮
- 该按钮实际是一个 `<a href={fileUrl} download={fileName}>` 直接下载链接
- 仅有 "Only View" 权限的用户可以通过此链接下载文件，完全绕过服务端的 `canDownload` 权限检查

**根本原因**：

`document-preview.tsx` 的 `switch` 语句 `default` 分支在文件类型不可预览时，直接渲染了一个指向 Supabase 签名 URL 的下载链接。该链接不经过任何权限校验：

```tsx
// 修复前：直接下载链接，绕过 canDownload 权限控制
default:
  return (
    <div>
      <FileIcon type={fileType} ... />
      <p>Preview not available for this file type</p>
      <p>{fileType}</p>
      <a href={fileUrl} download={fileName}>  // ← 权限绕过点
        <Button variant="outline">
          <Download /> Download to view
        </Button>
      </a>
    </div>
  );
```

而文档详情页 `doc/[slug]/page.tsx` 的 `DocumentActions` 组件通过服务端计算的 `canDownload` 属性控制下载按钮的可用性：

```tsx
// DocumentActions 组件：受权限控制的下载按钮
{canDownload ? (
  <Button onClick={handleDownload}>Download</Button>
) : (
  <Button disabled>Download</Button>  // Only View 用户看到禁用按钮
)}
```

两处下载入口的权限检查不一致：主按钮受 `canDownload` 控制，但预览区域的 fallback 下载链接完全不受控制。

### 权限影响范围分析

经全面审查，权限系统在电脑端和移动端的表现完全一致，不存在移动端特有的权限漏洞：

| 权限控制层 | 实现位置 | 设备无关性 |
|---|---|---|
| 路由保护（认证/角色/有效期） | `middleware.ts` | 服务端中间件，所有请求统一拦截 |
| 文档访问控制 | `doc/[slug]/page.tsx` SSR | 服务端渲染，权限判断在服务器执行 |
| 数据库行级安全（RLS） | Supabase 数据库层 | 数据库策略，与请求来源无关 |
| API 权限验证 | `api/users/route.ts` 等 | 服务端 API，统一检查 admin 角色 |

**唯一的问题**就是本次修复的 `document-preview.tsx` 中的 fallback 下载链接，同时影响电脑端和移动端。

## 解决方案与实施

### 1. 移除权限绕过的下载链接

将 `default` 分支中的 `<a>` 下载链接替换为友好的提示信息：

```diff
  default:
    return (
      <div className="flex flex-col items-center justify-center py-20 text-muted-foreground">
        <FileIcon type={fileType} className="h-16 w-16 mb-4 opacity-50" />
        <p className="text-sm">Preview not available for this file type</p>
        <p className="text-xs mt-1">{fileType}</p>
-       <a href={fileUrl} download={fileName} className="mt-4">
-         <Button variant="outline" className="gap-2">
-           <Download className="h-4 w-4" />
-           Download to view
-         </Button>
-       </a>
+       <p className="text-xs mt-3 text-muted-foreground/70">
+         Please use the Download button above if you have download permission.
+       </p>
      </div>
    );
```

### 2. 清理未使用的导入

移除下载链接后，`Button` 和 `Download` 导入不再被引用，一并清理：

```diff
  import dynamic from "next/dynamic";
  import { FileIcon } from "./file-icon";
- import { Button } from "@/components/ui/button";
- import { Download } from "lucide-react";
```

## 技术实现细节

### 权限控制流程

```
用户访问文档详情页
  ↓
服务端 SSR：计算 canDownload（admin/editor/owner → true，viewer → RPC 查询）
  ↓
DocumentActions 组件：根据 canDownload 渲染启用/禁用的 Download 按钮
  ↓
DocumentPreview 组件：根据文件类型渲染预览
  ↓
不可预览文件类型 → 显示"无法预览"提示 + 引导使用上方 Download 按钮
```

### 修改前后对比

| 场景 | 修改前 | 修改后 |
|---|---|---|
| Only View 用户访问不可预览文件 | 可通过 fallback 链接下载 | 仅看到提示信息，无法下载 |
| Download+ 用户访问不可预览文件 | 两个下载入口（主按钮 + fallback 链接） | 仅主按钮一个下载入口 |
| 未登录用户访问 | 无法到达此页面（中间件拦截） | 不变 |

## 经验教训

### 1. UI 降级策略的安全兜底规范
在 UI 组件中处理不可预览/不可用资源时，禁止提供绕过权限控制的直接操作入口（如未校验权限的下载链接）。应统一展示友好提示，引导用户通过受权限控制的主操作路径执行动作。

### 2. 多处操作入口的权限一致性
当一个页面存在多个功能入口时（如主操作按钮 + 预览区域 fallback），必须确保所有入口的权限检查逻辑一致。仅在服务端计算一次权限并向下传递（如 `canDownload` prop），避免各组件独立判断导致遗漏。

### 3. 代码审查关注点
在审查涉及权限控制的组件时，不仅要看主操作路径，还要关注 fallback/降级/错误处理路径中是否存在未受控的操作入口。
