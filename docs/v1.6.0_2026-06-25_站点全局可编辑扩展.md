## 任务目标

将现有的站点设置管理页面从仅支持首页 Hero 文案编辑，扩展为覆盖全站内容的可视化编辑中心。具体目标：

1. 首页 Hero 下方三个特性卡片（Multi-Format Support、Smart Organization、Access Control）支持管理员编辑标题、描述、图标，并可整体隐藏
2. 导航栏左上角 Logo 图标和站点名称支持管理员自定义
3. 底栏品牌描述和版权文字支持管理员编辑
4. 首页 Categories 区域支持管理员隐藏
5. 所有编辑支持实时预览，保存后立即生效

## 问题诊断与分析

### 问题 1: 首页内容定位与用户需求不匹配

**现象**：
- 首页硬编码了"Multi-Format Support"、"Smart Organization"、"Access Control"三个通用文档分享平台特性卡片
- 站点名称"DocShare"、Logo 图标、底栏描述文字均为代码常量，无法通过管理后台修改
- 用户希望将 DocShare 作为专用项目人员文档分享网站使用，而非通用文档分享平台

**根本原因**：
- 站点设置页面（v1.2.0 引入）仅覆盖 Hero 文案的 3 个 key，其余全站内容仍为前端硬编码
- 导航栏 Header 和底栏 Footer 组件直接引用 `APP_NAME` 常量，无动态数据来源
- 特性卡片和 Categories 区域没有可见性控制开关

### 问题 2: 全局品牌信息缺乏统一数据源

**现象**：
- 导航栏和底栏各自独立引用 `APP_NAME` 常量，无法统一管理
- 修改站点名称需要同时修改多个文件并重新部署

**根本原因**：
- 缺少一个全局的站点设置 Context，为 Header/Footer 等客户端组件提供统一的品牌信息

## 解决方案与实施

### 1. 数据库层扩展

复用现有 `site_settings` 表的 key-value 结构，新增 16 个设置项：

| Key | 默认值 | 说明 |
|-----|--------|------|
| `site_name` | DocShare | 导航栏/底栏站点名称 |
| `site_icon` | file-text | 导航栏 lucide 图标名 |
| `features_visible` | true | 特性卡片区域是否显示 |
| `feature_card_1_title/desc/icon` | Multi-Format Support / ... / file-text | 卡片 1 配置 |
| `feature_card_2_title/desc/icon` | Smart Organization / ... / folder-open | 卡片 2 配置 |
| `feature_card_3_title/desc/icon` | Access Control / ... / shield | 卡片 3 配置 |
| `categories_visible` | true | Categories 区域是否显示 |
| `footer_description` | A modern document... | 底栏品牌描述 |
| `footer_copyright` | (空) | 底栏版权文字（空则自动生成） |

通过 `INSERT ... ON CONFLICT DO NOTHING` 保证幂等性。

### 2. 全局站点设置 Context

创建 `SiteSettingsProvider`，在根布局中包裹，为所有客户端组件提供品牌信息：
- 加载 `site_name`、`site_icon`、`footer_description`、`footer_copyright` 四个全局 key
- 使用硬编码默认值作为 fallback，避免加载闪烁
- 暴露 `refresh()` 方法供管理员保存后刷新

### 3. 动态图标渲染工具

创建 `DynamicIcon` 组件和 `IconPicker` 选择器：
- `DynamicIcon`：通过字符串名称映射渲染 18 个 lucide 图标
- `IconPicker`：可视化网格选择器，支持选中高亮

### 4. 前端组件改造

- **Header**：从 `useSiteSettings()` 获取站点名称和图标，替代硬编码常量
- **Footer**：改为客户端组件，从 Context 获取品牌描述和版权文字
- **首页**：服务端查询所有 site_settings key，动态渲染特性卡片和 Categories 可见性

### 5. 管理员设置页重构

将 site-settings 页面从单 Tab 编辑重构为 5 个标签页：
- **Brand**：站点名称 + 图标选择 + 导航栏 Logo 预览
- **Hero**：保留原有功能（主标题、高亮标题、副标题）
- **Features**：显示/隐藏开关 + 3 张卡片的标题、描述、图标编辑 + 实时预览
- **Sections**：Categories 区域显示/隐藏开关
- **Footer**：品牌描述 + 版权文字编辑 + 底栏预览

每个 Tab 独立 Save / Restore Defaults 按钮。

## 技术实现细节

### 动态图标映射

```typescript
// src/lib/dynamic-icon.tsx
const ICON_MAP: Record<string, ComponentType<LucideProps>> = {
  "file-text": FileText,
  "folder-open": FolderOpen,
  shield: Shield,
  "book-open": BookOpen,
  globe: Globe,
  rocket: Rocket,
  // ... 共 18 个图标
};

export function DynamicIcon({ name, ...props }: DynamicIconProps) {
  const IconComponent = ICON_MAP[name] || FileText;
  return <IconComponent {...props} />;
}
```

### 全局 Context 架构

```typescript
// src/components/site-settings-provider.tsx
export function SiteSettingsProvider({ children }: { children: ReactNode }) {
  const [settings, setSettings] = useState<SiteSettings>(DEFAULTS);
  // 从 site_settings 表加载 site_name, site_icon, footer_description, footer_copyright
  // 使用硬编码默认值作为 fallback
  return (
    <SiteSettingsContext.Provider value={{ settings, loading, refresh }}>
      {children}
    </SiteSettingsContext.Provider>
  );
}
```

## 经验教训

### 1. Context 分层策略
全局品牌信息（site_name、site_icon、footer_*）通过 Context 提供给 Header/Footer 等客户端组件，而首页的特性卡片和 Categories 设置在服务端组件中直接查询数据库，避免不必要的客户端数据加载开销。

### 2. Write 工具追加问题
使用 Write 工具覆盖文件时，旧内容可能被追加到文件末尾。需要在写入后检查文件末尾是否有重复内容并清理。

### 3. 渐进增强设计
所有设置项都有硬编码默认值作为 fallback，即使数据库未执行 Migration 也不影响现有功能，保证了系统的健壮性。
