## 基本信息
- **日期**: 2026-06-25
- **类型**: 功能增强 + Bug 修复
- **涉及模块**: 权限系统、用户管理后台、文档详情页
- **状态**: ✅ 已完成

## 任务目标

1. 将权限模型从 3 级 (view/edit/manage) 扩展为 4 级，新增 `download` 权限级别，实现"仅浏览不可下载"的细粒度控制。
2. 修复管理员用户管理页面 Deactivate 按钮实际未生效的问题。
3. 新增管理员对用户账号的物理删除功能。

## 问题诊断与分析

### 问题 1: 权限粒度不足，缺少下载控制

**现象**：
- 原有权限模型仅有 `view` / `edit` / `manage` 三级，无法区分"仅可浏览"和"可浏览+可下载"场景。
- 所有有权限查看文档的用户都可以下载，无法满足"仅授权在线预览"的管理需求。

**根因**：
- 数据库枚举类型 `permission_type` 缺少 `download` 值。
- `has_document_permission()` 函数不支持层级权限判断（仅做精确匹配）。
- 文档详情页未做任何下载权限校验，所有已认证用户均可下载。

### 问题 2: Deactivate 按钮无效（状态不变）

**现象**：
- 管理员在 `/admin/users` 页面点击 Deactivate 按钮后，右下角 toast 弹出 "User Deactivated" 成功提示。
- 但用户 Status 列仍显示 "Active"，数据未实际更新。

**根因**：
- `toggleActive()` 函数使用浏览器端 Supabase 客户端（anon key）直接操作 `profiles` 表。
- `profiles` 表的 RLS 策略 `"Users update own profile"` 限制 `auth.uid() = id`，即用户只能更新自己的 profile。
- 管理员尝试停用其他用户时，RLS 静默拒绝（0 行受影响、无错误返回），代码误判为成功并显示 toast。

### 问题 3: 缺少用户物理删除功能

**现象**：
- 用户管理页面仅有 Deactivate/Activate 切换按钮，无法彻底删除用户账号。
- 需要手动到 Supabase Dashboard 操作，流程繁琐。

## 解决方案与实施

### 1. 权限模型 4 级扩展

#### 1.1 数据库迁移 `005_permission_download_level.sql`

新增 `download` 枚举值，权限层级为 `view(1) < download(2) < edit(3) < manage(4)`，高层级自动包含低层级权限。

```sql
-- 为 permission_type 枚举新增 'download' 值
ALTER TYPE permission_type ADD VALUE IF NOT EXISTS 'download' AFTER 'view';

-- 重写 has_document_permission() 实现层级权限判断
CREATE OR REPLACE FUNCTION has_document_permission(doc_id UUID, required_perm permission_type)
RETURNS boolean AS $$
-- 内部按 CASE 映射 view=1, download=2, edit=3, manage=4
-- 依次检查：admin bypass → 公开文档 view 级 → 用户直接权限 → 用户组权限 → 分类权限
$$ LANGUAGE plpgsql SECURITY DEFINER STABLE;
```

#### 1.2 文档详情页下载权限校验

在 `src/app/doc/[slug]/page.tsx` 服务端渲染阶段判断当前用户是否具备 `download` 级别及以上权限：

- Admin 和文档上传者：始终可下载
- 普通用户：检查显式权限（用户权限 / 用户组权限 / 分类权限）中的最高级别是否 ≥ `download(2)`

#### 1.3 前端下载按钮受控显示

`DocumentActions` 组件新增 `canDownload` prop：
- `canDownload = true`：正常显示 Download 按钮
- `canDownload = false`：按钮置灰、不可点击，tooltip 提示 "You only have view permission for this document"

#### 1.4 权限管理页 UI 更新

`/admin/permissions` 页面权限下拉选项更新为：
- Only View
- View & Download
- Edit
- Manage

列表展示同步使用语义化文案。

### 2. 用户管理功能修复与增强

#### 2.1 新建 API Route `/api/users`

采用服务端 admin client 绕过 RLS，包含两个端点：

**`PATCH /api/users`** — 切换用户激活状态
```typescript
// 使用 adminClient 直接更新 profiles.is_active
const adminClient = createAdminClient();
await adminClient.from("profiles").update({ is_active: !isActive }).eq("id", userId);
```

**`DELETE /api/users`** — 物理删除用户
```typescript
// 调用 auth.admin.deleteUser() 从 auth.users 表物理删除
// 级联删除 profiles 记录
await adminClient.auth.admin.deleteUser(userId);
await adminClient.from("profiles").delete().eq("id", userId);
```

两端点均含管理员身份验证 + 防止管理员自删。

#### 2.2 前端页面更新

`/admin/users` 页面新增：
- Deactivate/Activate 按钮改为调用 `PATCH /api/users`（走服务端 admin client）
- 新增红色 Delete 按钮（Trash2 图标），点击弹出确认弹窗
- 按钮 tooltip 提示：`Deactivate user` / `Activate user` / `Permanently delete user`

## 经验教训

### 1. Supabase 跨用户操作必须使用服务端 admin client
RLS 策略 `auth.uid() = id` 限制了用户只能修改自己的数据。管理员对其他用户的操作（如停用/删除）必须通过服务端 API route 使用 admin client（service role key）绕过 RLS，否则更新会被静默拒绝（0 行受影响、无错误）。

### 2. Supabase UPDATE 静默失败需额外检测
Supabase 的 `.update()` 在 RLS 拒绝时不返回 error，而是 `count: 0`。仅依赖 `error` 字段判断是否成功是不可靠的，建议检查 `data` 或 `count` 字段确认实际影响行数。

### 3. ALTER TYPE ADD VALUE 在事务中的注意事项
PostgreSQL 12+ 支持在事务块中执行 `ALTER TYPE ADD VALUE`，但旧版本不支持。Supabase 使用 PG 15+，可安全使用 `BEGIN/COMMIT` 包裹。
