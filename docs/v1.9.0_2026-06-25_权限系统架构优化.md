## 任务目标

对 DocShare 权限系统进行架构级优化，解决角色语义割裂、权限冗余无约束、管理 UX 不透明等核心矛盾。采用"角色基线 + 显式覆盖层"方案（方案 C），实现渐进式升级。

### 核心目标
1. 重写 `has_document_permission()` 函数，引入角色默认基线
2. 确保角色基线**仅影响操作权限**（download+），**不影响 RLS 可见性**
3. 添加权限表唯一约束，防止重复授权
4. 前端 canDownload 改用 RPC 统一判断
5. 权限授予改为 UPSERT 语义
6. 新增权限变更审计日志
7. 重构 Admin 权限管理页面 UI

## 问题诊断与分析

### 问题 1: 角色无权限含义
- Viewer/Editor 角色设定后，用户仍看不到任何私有文档
- 管理员需逐一手动授权，角色名称形同虚设

### 问题 2: 权限管理 UX 混淆
- User 下拉框混合展示所有用户，无法区分角色
- 无法预览某用户的最终有效权限
- 权限冲突（文档级 vs 分类级）不可见

### 问题 3: 权限冗余无约束
- 同一用户对同一文档可重复授予不同级别权限
- 无唯一性约束，数据产生冗余记录

### 问题 4: canDownload 前后端逻辑不同步
- 前端直接查询 `permissions` 表，未覆盖用户组权限和分类权限
- 与数据库 `has_document_permission()` 函数的判断逻辑存在差异

## 解决方案与实施

### Phase 1: 数据库迁移 — `009_permission_role_baseline.sql`

**文件**: `supabase/migrations/009_permission_role_baseline.sql`

包含 5 个部分：

#### Part 1: 重写 `has_document_permission()` 函数
- Admin 短路检查
- 公开文档隐式 view 权限
- 显式用户权限聚合（文档级 + 分类级）
- 用户组权限聚合（文档级 + 分类级）
- **角色基线仅在 `perm_level > 1` 时生效**（关键设计：不影响 RLS 可见性）

```sql
-- Step 6: Role baseline — ONLY for operations beyond 'view'
IF perm_level > 1 THEN
  SELECT role INTO user_role_text FROM public.profiles WHERE id = auth.uid();
  role_base_level := CASE user_role_text
    WHEN 'editor' THEN 2  -- Editor baseline = download
    ELSE 0               -- Viewer has no baseline
  END;
  user_max_level := GREATEST(user_max_level, role_base_level);
END IF;
```

#### Part 2: 完整重建 4 条 RLS 策略
- `DROP POLICY IF EXISTS` 全部 4 条策略后重建
- 修复 `CASCADE` 仅删除依赖函数的策略（SELECT/UPDATE），不删除 INSERT/DELETE 策略的问题

```sql
DROP POLICY IF EXISTS "Public documents viewable" ON public.documents;
DROP POLICY IF EXISTS "Editors modify own docs" ON public.documents;
DROP POLICY IF EXISTS "Editors can insert" ON public.documents;
DROP POLICY IF EXISTS "Admins can delete" ON public.documents;
```

#### Part 3: 清理重复权限记录
- 使用 `ROW_NUMBER()` 窗口函数，按权限等级降序 + 创建时间降序排序
- 保留最高权限记录（而非最新记录）

#### Part 4: 创建唯一约束
- 使用 `ALTER TABLE ADD CONSTRAINT UNIQUE`（而非 `CREATE UNIQUE INDEX`）
- 兼容 PostgREST 的 `upsert/onConflict` 语义

```sql
ALTER TABLE public.permissions
  ADD CONSTRAINT uq_user_doc_perm UNIQUE (user_id, document_id);
ALTER TABLE public.permissions
  ADD CONSTRAINT uq_user_cat_perm UNIQUE (user_id, category_id);
```

#### Part 5: 扩展审计日志枚举
```sql
ALTER TYPE access_action ADD VALUE IF NOT EXISTS 'permission_grant';
ALTER TYPE access_action ADD VALUE IF NOT EXISTS 'permission_revoke';
```

### Phase 1: 前端 — canDownload 改用 RPC

**文件**: `src/app/doc/[slug]/page.tsx`

```typescript
if (profile?.role === "admin" || document.uploaded_by === user.id) {
  canDownload = true;
} else if (profile?.role === "editor") {
  canDownload = true; // Editor baseline: auto-download on visible docs

  canDownload = hasDownload === true;

```

### 踩坑记录

| 问题 | 原因 | 修复 |
|------|------|------|
| `policy "Editors can insert" already exists` | `CASCADE` 仅删除依赖该函数的策略，INSERT/DELETE 不依赖 `has_document_permission()` | 显式 `DROP POLICY IF EXISTS` 所有 4 条策略 |

## 经验教训

### 1. CASCADE 删除的范围限制
`DROP FUNCTION ... CASCADE` 只删除**依赖该函数的对象**。如果 RLS 策略的 USING/WITH CHECK 表达式中未引用该函数，则不会被级联删除。INSERT 和 DELETE 策略通常不依赖权限检查函数，需手动 DROP。

### 2. PostgREST upsert 的约束识别
PostgREST 的 `onConflict` 参数只识别 `pg_constraint` 表中的 UNIQUE CONSTRAINT，不识别独立创建的 UNIQUE INDEX。使用 `ALTER TABLE ADD CONSTRAINT ... UNIQUE` 而非 `CREATE UNIQUE INDEX`。

### 3. 角色基线与 RLS 的分离设计
角色基线不能直接作用于 RLS SELECT 策略，否则 Editor 会自动看到所有私有文档（因为 `has_document_permission(id, 'view')` 会因 role_base_level=2 >= perm_level=1 返回 true）。关键设计：仅在 `perm_level > 1` 时应用角色基线。

### 4. 迁移脚本的幂等性
使用 `BEGIN/COMMIT` 事务包裹确保原子性；`IF NOT EXISTS` / `IF EXISTS` 确保重复执行不报错。
