## 任务目标

修复用户反馈的两个认证状态相关问题：
1. **Bug**：用户退出登录后返回首页，点击右上角 Sign In / Sign Up 按钮无任何响应，必须刷新页面才能恢复
2. **增强**：用户打开多个浏览器 Tab 时，在其中一个 Tab 退出登录，其它 Tab 需保持状态同步

## 问题诊断与分析

### 问题 1: 登出后 Sign In/Sign Up 按钮失效

**现象**：
- 用户退出登录后，Header 正确切换为 Sign In / Sign Up 按钮（React 状态已更新）
- 但点击按钮无任何响应，无报错，页面停留在当前 URL 不变
- 手动刷新页面后按钮恢复正常

**根本原因**：

Next.js App Router 的 Router Cache（客户端路由缓存）在登出后未被刷新。

登出流程：
1. `supabase.auth.signOut()` 清除 Session Cookie
2. `setUser(null)` / `setProfile(null)` 更新 React 状态 → Header 正确切换为未登录 UI
3. ❌ **缺失** `router.refresh()` → Router Cache 未失效

当用户点击 `<Link href="/login">` 时，Next.js 中间件执行以下逻辑：

```typescript
// src/lib/supabase/middleware.ts
// Auth routes - redirect to home if already logged in
const authPaths = ["/login", "/register"];
const isAuthPath = authPaths.some((path) =>
  request.nextUrl.pathname.startsWith(path)
);
if (isAuthPath && user) {
  const url = request.nextUrl.clone();
  url.pathname = "/";
  return NextResponse.redirect(url);
}
```

由于 Router Cache 中仍缓存着"用户已登录"的响应，中间件认为用户仍处于登录状态，将 `/login` 重定向回 `/`。用户本身就在首页，重定向到 `/` 看起来就像"按钮没反应"。

### 问题 2: 多 Tab 状态不同步

**现象**：
- 用户在 Tab A 退出登录，Tab B/C 的 Header 仍显示用户头像和下拉菜单
- 在 Tab B 点击受保护页面（如 /admin）时，才被中间件拦截跳转到登录页

**根本原因**：

Supabase JS v2.x 内部通过 `BroadcastChannel` API 在多 Tab 之间广播认证事件。当 Tab A 调用 `signOut()` 时，其它 Tab 的 `onAuthStateChange` 监听器确实会收到 `SIGNED_OUT` 事件，React 状态也会更新。

但 `onAuthStateChange` 回调中缺少 `router.refresh()` 调用，导致：
- Header 的 React 状态正确更新（显示 Sign In/Sign Up）
- Router Cache 未刷新 → 受保护页面内容不会重新渲染
- 后续导航行为异常

## 解决方案与实施

### 核心修复：补充 `router.refresh()` 调用

在 `src/components/auth-provider.tsx` 中做了两处关键修改：

**修复 1：`signOut` 函数末尾添加 `router.refresh()`**

```typescript
const signOut = async () => {
  // Log the logout event before signing out
  if (user) {
    await supabase.from("access_logs").insert({
      user_id: user.id,
      action: "logout" as const,
    });
  }
  await supabase.auth.signOut();
  setUser(null);
  setProfile(null);
  router.refresh(); // ← 新增：刷新当前 Tab 的 Router Cache
};
```

**修复 2：`onAuthStateChange` 中监听 `SIGNED_OUT` 事件**

```typescript
const {
  data: { subscription },
} = supabase.auth.onAuthStateChange(async (event, session) => {
  setUser(session?.user ?? null);
  if (session?.user) {
    await fetchProfile(session.user.id);
  } else {
    setProfile(null);
  }
  // When signed out (e.g. from another tab via BroadcastChannel), refresh router cache
  if (event === "SIGNED_OUT") {
    router.refresh(); // ← 新增：其它 Tab 登出时也刷新 Router Cache
  }
});
```

## 技术实现细节

### 多 Tab 同步完整流程

```
Tab A (用户点击 Sign Out)     →     Tab B, C, D (自动同步)
         │                                    │
    signOut()                     onAuthStateChange 触发
         │                          (Supabase BroadcastChannel)
         ▼                                    ▼
  1. 清除 session cookie          1. event = "SIGNED_OUT"
  2. setUser(null)                2. setUser(null) → Header 更新
  3. setProfile(null)             3. setProfile(null)
  4. router.refresh()             4. router.refresh() ← 新增
```

### 技术要点

| 层面 | 机制 | 说明 |
|------|------|------|
| **跨 Tab 通信** | Supabase JS v2.x `BroadcastChannel` API | 浏览器原生 API，同域名下多 Tab 自动广播 |
| **React 状态同步** | `onAuthStateChange` 监听器 | 各 Tab 收到事件后独立更新 `user`/`profile` |
| **Router Cache 刷新** | `router.refresh()` | 强制 Next.js 重新请求服务端组件，使中间件重新评估认证状态 |

### 闭包问题规避

在 `onAuthStateChange` 回调中，`user` 变量因 `useEffect` 依赖数组为空而捕获的是初始值 `null`。因此不能使用 `previousUser` 判断条件，改为直接监听 `event === "SIGNED_OUT"` 事件，避免闭包陷阱。

## 经验教训

### 1. Next.js App Router 登出必须刷新 Router Cache
在 Next.js App Router 中，`router.refresh()` 是登出流程的**必要步骤**。仅更新 React 状态不足以使后续导航正确工作，因为 Router Cache 中可能缓存着"用户已登录"时的服务端响应。

### 2. `useEffect` 闭包陷阱
在 `useEffect` 内使用 `useState` 变量时，由于依赖数组为空，闭包捕获的是初始值。需要通过事件参数（如 `event`）而非状态变量来判断条件，避免逻辑错误。

### 3. Supabase 自带跨 Tab 同步能力
`@supabase/supabase-js` v2.x 内置了 `BroadcastChannel` 跨 Tab 通信机制，无需额外开发。只需确保 `onAuthStateChange` 回调中正确处理所有认证事件即可。
